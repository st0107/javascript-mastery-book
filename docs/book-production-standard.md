## Explanation Contract

For each important concept, the author must answer:

- What is the definition?
- Why does it exist?
- What problem does it solve?
- How does it work internally?
- What are the execution steps?
- What does memory look like?
- What visual model helps explain it?
- What is a realistic example?
- What is a production example?
- How should it be explained in interviews?
- What mistakes do engineers make?
- What edge cases matter?
- What are the performance trade-offs?
- When should it not be used?
- What are the best practices?

## Code Contract

Every code example must:

- Run independently.
- Use modern JavaScript.
- Avoid global side effects except deliberate console output.
- Include expected output in a comment.
- Include time and space complexity when algorithmic behavior matters.
- Prefer realistic domain examples over isolated toy snippets.

## Diagram Contract

Mermaid diagrams must be stored in `diagrams/` and embedded or linked from chapters. Diagrams should be used for execution flow, memory, scheduling, inheritance, browser architecture, rendering, module loading, and engine pipelines.
